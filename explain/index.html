What is cloud security?
Most people think getting hacked means someone writes clever exploit code. In reality, most cloud breaches happen because someone left a door open. A public S3 bucket. An admin account with no password protection. A database exposed to the entire internet. Cloud security is the work of finding and closing those doors.
🪣
S3 Buckets
S3 is AWS cloud storage. Companies store everything there — backups, user uploads, application files. If a bucket is accidentally set to public, anyone on the internet can download those files. This has caused some of the biggest data breaches in history.
🔑
IAM (Identity and Access Management)
IAM controls who can do what in your AWS account. The most common mistake is giving users far more permissions than they need. If one of those users gets hacked, the attacker inherits all their permissions.
🔒
Security Groups
Security groups are cloud firewalls. They define which network traffic is allowed. Leaving a port like SSH or RDP open to the entire internet is like leaving your front door unlocked — attackers scan for this constantly.
📋
CloudTrail
CloudTrail records every action taken in your AWS account. Who deleted that file? Who changed the firewall rule? Without CloudTrail enabled, you have no way to answer those questions after an incident.
📏
CIS Benchmark
The Center for Internet Security publishes a free checklist of security controls specifically for AWS. It is what security teams and auditors use to assess cloud environments. Passing it means you have covered the basics.
⚖️
Least Privilege
The principle of least privilege means every user, service and application should have only the minimum access needed to do its job — nothing more. It limits the damage if something goes wrong.
Who is responsible?
The most important concept in cloud security. AWS secures the physical infrastructure. You are responsible for everything you build and configure on top of it. This is called the Shared Responsibility Model.

AWS is responsible for

"Security OF the cloud"
  • 🏢 Physical data centres
  • 🌐 Network infrastructure
  • 💻 Server hardware
  • 🔧 Hypervisor and virtualisation
  • ⚡ Power and cooling
  • 🛡️ Managed service security
AWS has never been hacked at the infrastructure level.

You are responsible for

"Security IN the cloud"
  • 🪣 What you store in S3
  • 🔑 Who has IAM access
  • 🔒 Your security group rules
  • 📋 Whether logging is on
  • 💻 Your application code
  • 🔐 Your data encryption choices
Most cloud breaches happen here.
Real example: In 2019, Capital One suffered a breach affecting over 100 million customers. It was not because AWS was hacked. It was because of a misconfigured web application firewall that the attacker exploited to access S3 buckets that should have been private. AWS did their job. The customer did not configure their side correctly.

CIS Level 1 checklist
The CIS AWS Foundations Benchmark is the standard auditors use. Check off each control to see your coverage. In a real engagement, every unchecked item is a finding.
0 / 10 controls passed
  • MFA enabled on root account
    The root account has unrestricted access to everything in your AWS account. If it gets compromised without MFA, you have no fallback.
  • No access keys on root account
    Root access keys are extremely dangerous. If they are leaked, an attacker has full control of your account. Delete them and use IAM users instead.
  • CloudTrail enabled in all regions
    Without CloudTrail, you have no record of who did what. Attackers know this and sometimes disable it first. Enable it everywhere and protect the logs.
  • No S3 buckets are publicly accessible
    Public S3 buckets are responsible for some of the largest data breaches in history. Enable Block Public Access on every bucket and at the account level.
  • SSH (port 22) not open to 0.0.0.0/0
    SSH open to the entire internet is one of the most scanned configurations on the internet. Restrict it to your IP or a VPN.
  • Access keys rotated within 90 days
    Old access keys are a risk. The longer a key exists, the more chance it has been leaked somewhere. Rotate them every 90 days maximum.
  • MFA enabled for all IAM console users
    Every human user who logs into the AWS console should have MFA. Passwords alone are not enough.
  • No inline policies with full admin access
    Giving a user AdministratorAccess is almost always wrong. Apply least privilege: give users only what they need for their specific job.
  • S3 bucket logging enabled
    Just like CloudTrail, bucket access logs tell you who accessed what and when. Essential for incident investigation.
  • Unused credentials disabled within 90 days
    Dormant accounts are a risk. An attacker who finds old credentials can use them months later. Disable anything that has not been used in 90 days.
